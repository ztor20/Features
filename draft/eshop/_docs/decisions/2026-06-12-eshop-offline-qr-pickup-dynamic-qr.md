# 2026-06-12 — [Ztor 2.0] eShop fulfillment: offline QR-code pickup only (no logistics, no refunds yet); Dynamic QR required

*Status: DRAFT — needs review*

## Decision
目前 Store **不對接物流**，只支援**線下 QR Code 提貨**。Creator Studio 需新增**訂單清單**及 **QR Code 掃碼驗證**功能。**退款功能暫不開發**（Store 現階段也不支援）。Shipping 初期可 offline 處理，但需安排物流供應商整合（目前 Ninja Van 針對東南亞，需另找香港/台灣物流商）。

QR Code 機制：本質是一個帶 Token/ID 的連結，由 **JavaScript Library 生成，不需第三方服務**；目前系統僅支援 Fixed QR Code，**需補強為 Dynamic QR Code（每 5–30 秒 refresh）**，以防截圖轉發的安全問題。不需集中管理系統，每個產品各自呼叫同一個 function 生成；Operator 只需驗證 ID 是否存在，不需儲存 QR Code 本身。未來 QR Code 也將用於 Pokémon-Go 類型功能（如與星巴克合作掃碼獲得 Loyalty Points）。

(English) The Store **does not integrate logistics** for now — it supports **offline QR-code pickup only**. Creator Studio needs an **order list** and **QR-code scan-verification**. **Refunds are not built yet** (the Store itself doesn't support them at this stage). Shipping can be handled offline initially but needs a logistics-provider integration (Ninja Van covers SEA; a HK/Taiwan carrier still needs sourcing). QR codes are token/ID-bearing links generated by a **JS library with no third-party service**; the system only does Fixed QR today and **must be upgraded to Dynamic QR (refresh every 5–30s)** to prevent screenshot-resharing. No central QR store is needed — each product calls the same generator function; the operator only validates the ID exists. QR will later power Pokémon-Go-style features (e.g. a Starbucks scan-for-Loyalty-Points tie-in).

## Why
線下提貨 + 動態 QR 在不建物流整合的前提下完成核心交付流程，並用每 5–30 秒刷新的 Dynamic QR 防止截圖轉發盜用。Offline pickup + dynamic QR delivers the core fulfillment loop without building logistics integration; the 5–30s refresh closes the screenshot-resharing security hole.
