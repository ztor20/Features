How to take the working demo to a production Ztor feature. Green = already proven in the demo; amber = what the dev team builds to productionize. The realtime core (Ably + host-authoritative sync) carries over unchanged. v2 folds in the two 2026-06-22 meetings: ops-granted hosts only, an ops-assigned content pool the host picks from in-room, a 1,000-seat cap, host-presence handling, confirmation emails, and watch-party orders counted toward total orders.

Core principle — host-authoritative sync. Only the host's player emits playback events (play / pause / seek). Every viewer is a passenger that replays them. There is no central media server doing the syncing — Ably is a thin pub/sub bus, and the video bytes come straight from the CDN to each client. This keeps the build small: the "magic" is ~3 channels and a snapshot-on-join handshake, not a streaming backend.
What changed after the 2026-06-22 meetings. The demo proved the sync mechanism; these decisions reshape the room model and add the back office.
  • Hosts are ops-granted only — no open self-serve. A back-office capability decides who can host.
  • Video is not fixed per room. Ops assigns a host a pool of titles + specific episodes; the host picks/switches among them in-room (marathon-friendly). Room record holds the host + allowed pool, not a single hard-coded videoId.
  • Host must stay present — host controls playback, so a disconnect drops the room into a host_away (paused) state until they return or ops ends it.
  • Capacity target ~1,000 (advertised ~400–500); enforced atomically before issuing a ticket/token.
  • Tickets are host-set (free or POPCORN) on the front end; each counts as an Order toward total orders. No replay; refunds are manual (ops marks refunded, CS re-credits).
  • Confirmation emails on party creation + ticket purchase (+ start reminder). Live chat is ephemeral (not retained after the room ends).

Sync protocol — host-authoritative
The only non-obvious part of the build. Two flows: live control (left→right propagation of host actions) and late-join catch-up (snapshot handshake). Everything rides the video-{room} channel. v2: the snapshot's videoId is now an episodeId — when the host switches episodes from the granted pool, the same status handshake re-broadcasts and every viewer adopts the new episode. If the host disconnects, the room goes host_away and playback holds for everyone until they reconnect (only the host emits events).

Demo → Production: what the dev team changes
The demo proves the mechanism end-to-end. These are the deltas to make it a real Ztor feature — most are "reuse an existing Ztor service," not net-new infrastructure.
ConcernDemo (shipped)Production for Ztor

Content selection 3 hard-coded clips; host picks one at create time Ops assigns a title+episode pool; host picks/switches episodes in-room (marathon-friendly), restricted to the grant.
Capacity Unbounded; whoever has the link joins Per-room cap (~1,000) enforced atomically before a ticket/token issues; separate advertised number is display-only.
